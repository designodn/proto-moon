# Обновление контента дизайнерами

Раздел **«Обновление контента»** на развилочной (`/content`) даёт дизайнерам всё
для самостоятельного наполнения прототипа:

1. **Редактировать контент** — кнопка на Google-таблицу.
2. **Загрузить медиа** — свои картинки и клипы заливаются в облако, страница
   отдаёт готовый URL → дизайнер вставляет его в нужную ячейку таблицы.
3. **Инструкция** — короткий чек-лист.

Внешние картинки по-прежнему задаются ссылкой в таблице. Новое — только путь для
**своих** файлов, у которых ссылки нет.

## Как это устроено

- Файлы уходят в **S3-бакет** (Yandex Object Storage), а не в репозиторий.
- Картинки по желанию сжимаются в WebP (≤1600px) — галочка на каждом файле; клипы
  заливаются как есть.
- Имя файла — контент-адресное (хэш содержимого), как и у остальных ассетов.
- `media-cache` (`scripts/lib/media-cache.mjs`) **не качает** ссылки на наш бакет в
  репо (passthrough по `UPLOADS_PUBLIC_BASE`).

### Облачный режим пайплайна (dual-mode)

`media-cache` работает в двух режимах — выбор автоматический:

- **env `UPLOADS_*` заданы → облачный режим.** Скачанные из таблицы картинки
  заливаются в бакет под путём `assets/<раздел>/<файл>`, в данные пишется
  публичный URL. Репозиторий **не растёт**.
- **env не заданы → дисковый режим (как раньше).** Картинки качаются в `assets/`.
  Так работает локальная разработка без ключей.

Это значит: код **обратно-совместим**. Пока не задашь env, всё ведёт себя как
сегодня. Облако «включается» переменными окружения — и так же выключается.

### Что переезжает, а что нет

- **В облако:** всё, что управляется манифестами `data/*-media.json` (ленты,
  аватары, клипы, подарки, профиль, сториз и т.д.) — то, что наполняют дизайнеры.
- **Остаётся в репо:** UI-иконки и мелкая статика (`assets/icons`, lottie, бейджи,
  lockscreen) — это часть интерфейса, не контент.
- **Пока в репо (можно перенести позже):** тяжёлый «вшитый» статик
  `assets/embedded`, `assets/koleso`, `assets/today`, `assets/new-vision` —
  захардкожен в конкретных экранах, требует правки ссылок вручную.

## Переменные окружения

Задаются на хостинге, где крутится `server.mjs` (там же, где развилочная):

| Переменная | Обязательна | Назначение |
|---|---|---|
| `UPLOADS_BUCKET` | да | имя бакета для загрузок |
| `UPLOADS_ACCESS_KEY_ID` | да | ключ S3 (можно переиспользовать `AWS_ACCESS_KEY_ID`) |
| `UPLOADS_SECRET_ACCESS_KEY` | да | секрет S3 (можно `AWS_SECRET_ACCESS_KEY`) |
| `UPLOAD_PASSWORD` | желательно | общий пароль на загрузку (без него — открыто) |
| `UPLOADS_PUBLIC_BASE` | желательно | публичная база URL, напр. `https://<bucket>.storage.yandexcloud.net/` |
| `UPLOADS_ENDPOINT` | нет | по умолчанию `https://storage.yandexcloud.net` |
| `UPLOADS_REGION` | нет | по умолчанию `ru-central1` |
| `UPLOADS_PREFIX` | нет | префикс ключей в бакете (по умолчанию пусто) |
| `SHEET_EDIT_URL` | нет | ссылка для кнопки «Редактировать» (по умолчанию рабочая таблица) |

`UPLOADS_PUBLIC_BASE` должен совпадать с тем, как реально отдаётся бакет, —
именно по этому префиксу работает passthrough.

### Боевой деплой (VM на Yandex Cloud): через секреты SourceCraft

На проде `/opt/proto-moon/.env` **пересоздаётся из секретов SourceCraft на каждом
деплое** (`.sourcecraft/ci.yaml`), поэтому править его руками по SSH бесполезно —
затрётся. Чтобы включить облако:

1. SourceCraft → репозиторий-зеркало → **Settings → Secrets** → добавь:
   - `UPLOADS_ACCESS_KEY_ID` — идентификатор статического ключа бакета;
   - `UPLOADS_SECRET_ACCESS_KEY` — секретный ключ;
   - `UPLOAD_PASSWORD` — пароль на страницу `/content` (по желанию).
2. Имя бакета и публичная база зашиты в `ci.yaml` литералом
   (`proto-design-okds-uploads`) — отдельными секретами не нужны.
3. Залей в `main` (любой push/merge) → CI запишет `.env` и пересоберёт контейнер.
4. Проверь: `docker compose exec web node scripts/check-bucket.mjs` на VM.

Для **локальной** разработки переменные задаются обычным окружением/`.env` рядом
с `server.mjs` — таблица выше.

## Настройка бакета (Yandex Object Storage)

1. В консоли Yandex Cloud создай **отдельный бакет** под загрузки (напр.
   `proto-moon-uploads`), доступ на чтение — **публичный**.
2. Создай статический ключ доступа для сервисного аккаунта с правами на запись в
   этот бакет → `UPLOADS_ACCESS_KEY_ID` / `UPLOADS_SECRET_ACCESS_KEY`.
3. Пропиши переменные окружения из таблицы выше.
4. Поставь `UPLOAD_PASSWORD` и раздай его дизайнерам.

## Доступ для дизайнеров

Дизайнерам **не нужны** ключи бакета — они заливают через страницу `/content`,
зная только `UPLOAD_PASSWORD`. Доступ отзывается сменой пароля. Если позже нужен
персональный доступ — заведи отдельные пароли/прокси, пока для 1–8 человек общего
пароля достаточно.

## Перенос существующего контента в облако

**Ничего из репозитория не удаляется.** Существующие файлы в `assets/` остаются
как страховка: если с бакетом не сложится — просто не задавай env, и сайт работает
с локальными копиями, как сейчас. Удаление файлов из git — отдельный
**необязательный** шаг, который делается потом, вручную, когда сама убедишься, что
облако работает.

### Шаг 1. Создай бакет и задай env (см. выше)

### Шаг 2. Залей существующие картинки в бакет (разово)

```bash
node scripts/migrate-assets.mjs --dry-run   # посмотреть, что зальётся
node scripts/migrate-assets.mjs             # залить контент-картинки
```

Скрипт берёт всё из манифестов `data/*-media.json` и кладёт в бакет под теми же
путями (`assets/<раздел>/<файл>`), которые пайплайн отдаёт в облачном режиме.
Важно для «мёртвых» источников: те картинки, что уже не пере-скачать, переносятся
из локальных копий.

### Шаг 3. Клипы

```bash
node scripts/migrate-clips.mjs              # зальёт MP4, напечатает URL
```

Впиши напечатанные URL в столбец **«видео»** строк листа «Клипы» (по id:
`clip-1.mp4` → строка `clip-1`), затем `node scripts/fetch-clips.mjs --force`.

### Шаг 4. Проверь и (по желанию) убери из git — потом

Включи env для пайплайна, прогони синк, открой сайт — картинки должны грузиться из
бакета. Когда убедишься, что всё ок, можно (не обязательно!) перестать
коммитить контент-папки:

```bash
git rm -r --cached assets/people assets/q3 assets/tribune assets/activity \
  assets/activity-pins assets/around-you assets/profile assets/gifts \
  assets/marathon assets/stories assets/clips
# и добавь эти пути в .gitignore
```

История git сохранит старые файлы (полный clone всё ещё их тянет). Полная чистка
истории — отдельная операция (`git filter-repo`), ломает существующие клоны.

## Скелетоны при загрузке

Чтобы сетевые картинки не «прыгали» пустыми рамками, на лентах включён шиммер-
плейсхолдер: `components/image-skeleton.js` вешает класс `.__state-loading`
(шиммер дизайн-системы) на контейнер картинки, пока она грузится, и снимает по
`load`/`error`. Подключён на основных лентах (NV, Q3, Трибуна, Активити, Клипы,
Профиль, Марафон, Подарки). Новой странице достаточно одной строки:

```html
<script src="components/image-skeleton.js" defer></script>
```
