# Обновление контента дизайнерами

Раздел **«Обновление контента»** на развилочной (`/content`) даёт дизайнерам всё
для самостоятельного наполнения прототипа:

1. **Редактировать контент** — кнопка на Google-таблицу.
2. **Загрузить медиа** — свои картинки и клипы заливаются в облако, страница
   отдаёт готовый URL → дизайнер вставляет его в нужную ячейку таблицы.
3. **Инструкция** — короткий чек-лист.

Внешние картинки по-прежнему задаются ссылкой в таблице. Новое — только путь для
**своих** файлов, у которых ссылки нет.

## Как это устроено

- Файлы уходят в **отдельный S3-бакет** (Yandex Object Storage), а не в репозиторий.
- Картинки по желанию сжимаются в WebP (≤1600px) — галочка на каждом файле; клипы
  заливаются как есть.
- Имя файла — контент-адресное (хэш содержимого), как и у остальных ассетов.
- `media-cache` (`scripts/lib/media-cache.mjs`) **не качает** ссылки на наш бакет в
  репо (passthrough по `UPLOADS_PUBLIC_BASE`) — поэтому репозиторий не растёт.
  Чужие CDN-ссылки кэшируются как и раньше.

## Переменные окружения

Задаются на хостинге, где крутится `server.mjs` (там же, где развилочная):

| Переменная | Обязательна | Назначение |
|---|---|---|
| `UPLOADS_BUCKET` | да | имя бакета для загрузок |
| `UPLOADS_ACCESS_KEY_ID` | да | ключ S3 (можно переиспользовать `AWS_ACCESS_KEY_ID`) |
| `UPLOADS_SECRET_ACCESS_KEY` | да | секрет S3 (можно `AWS_SECRET_ACCESS_KEY`) |
| `UPLOAD_PASSWORD` | желательно | общий пароль на загрузку (без него — открыто) |
| `UPLOADS_PUBLIC_BASE` | желательно | публичная база URL, напр. `https://<bucket>.storage.yandexcloud.net/` |
| `UPLOADS_ENDPOINT` | нет | по умолчанию `https://storage.yandexcloud.net` |
| `UPLOADS_REGION` | нет | по умолчанию `ru-central1` |
| `UPLOADS_PREFIX` | нет | префикс ключей в бакете (по умолчанию пусто) |
| `SHEET_EDIT_URL` | нет | ссылка для кнопки «Редактировать» (по умолчанию рабочая таблица) |

`UPLOADS_PUBLIC_BASE` должен совпадать с тем, как реально отдаётся бакет, —
именно по этому префиксу работает passthrough.

## Настройка бакета (Yandex Object Storage)

1. В консоли Yandex Cloud создай **отдельный бакет** под загрузки (напр.
   `proto-moon-uploads`), доступ на чтение — **публичный**.
2. Создай статический ключ доступа для сервисного аккаунта с правами на запись в
   этот бакет → `UPLOADS_ACCESS_KEY_ID` / `UPLOADS_SECRET_ACCESS_KEY`.
3. Пропиши переменные окружения из таблицы выше.
4. Поставь `UPLOAD_PASSWORD` и раздай его дизайнерам.

## Доступ для дизайнеров

Дизайнерам **не нужны** ключи бакета — они заливают через страницу `/content`,
зная только `UPLOAD_PASSWORD`. Доступ отзывается сменой пароля. Если позже нужен
персональный доступ — заведи отдельные пароли/прокси, пока для 1–8 человек общего
пароля достаточно.

## Перенос существующих клипов из репо

Сейчас 6 MP4 лежат в `assets/clips/` (~37 МБ). Разовый перенос:

```bash
node scripts/migrate-clips.mjs        # зальёт в бакет, напечатает URL
```

Затем:
1. Впиши напечатанные URL в столбец **«видео»** соответствующих строк листа «Клипы»
   (сопоставляй по id: `clip-1.mp4` → строка с id `clip-1`).
2. Пересобери: `node scripts/fetch-clips.mjs --force`.
3. Удали локальные файлы: `git rm assets/clips/*.mp4 && git commit`.

История git при этом сохранит старые файлы (полный clone всё ещё их тянет).
Рабочая копия и деплой полегчают сразу; полная чистка истории — отдельная
операция (`git filter-repo`), ломает существующие клоны.
